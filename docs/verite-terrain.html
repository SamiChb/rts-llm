1. Pourquoi une vérité terrain ?

RTS-LLM est un outil de sélection de tests de régression : à partir d'un diff Git, il prédit quels scénarios BDD doivent être ré-exécutés. Pour évaluer cette prédiction, il faut un oracle qui dit, pour le même diff, quels scénarios auraient réellement dû être ré-exécutés.

Cet oracle s'appelle la vérité terrain. Sa fonction n'est pas d'être un outil RTS concurrent, mais un étalon de mesure :

Diff Git d'un commit | +-----------+----------+ | | v v RTS-LLM (statique + LLM) Vérité terrain (JaCoCo) -> S = {sc_1, sc_5} -> G = {sc_1, sc_3} | | +----------+-----------+ | v Métriques : Précision = |S inter G| / |S| Rappel = |S inter G| / |G|

La vérité terrain est par définition correcte au regard de l'exécution réelle des tests ; les écarts mesurés (faux positifs, faux négatifs de RTS-LLM) reflètent les limites de la prédiction, pas de l'oracle.

2. Principe : couverture dynamique avec JaCoCo

La méthodologie repose sur l'observation suivante : un scénario BDD est impacté par une modification si, et seulement si, son exécution traverse au moins une des méthodes modifiées.

La couverture dynamique répond exactement à cette question : pour chaque scénario, on enregistre les méthodes effectivement exécutées lors de son passage, en instrumentant le bytecode avec JaCoCo.

L'équation centrale de la vérité terrain s'écrit alors :

G(diff) = { s ∈ Scénarios | Couverture(s) ∩ MéthodesModifiées(diff) ≠ ∅ }

où :

  • Couverture(s) est l'ensemble des méthodes exécutées par le scénario s, mesuré une fois pour toutes par JaCoCo dans la version de référence du code ;
  • MéthodesModifiées(diff) est l'ensemble des méthodes que le diff Git modifie, calculé par croisement entre les hunks du diff et les plages de méthodes issues de l'AST.

3. Pipeline de calcul, étape par étape

3.1. Préparation du projet cible

Le projet à évaluer doit avoir le plugin jacoco-maven-plugin configuré, ou bien être lancé avec l'agent JaCoCo injecté à la ligne de commande (mode --no-pom-changes, qui évite de polluer l'historique Git). 

// Dans le pom.xml du projet cible (mode permanent)
<plugin>
  <groupId>org.jacoco</groupId>
  <artifactId>jacoco-maven-plugin</artifactId>
  <version>0.8.11</version>
  <executions>
    <execution><goals><goal>prepare-agent</goal></goals></execution>
  </executions>
</plugin>

3.2. Étape 1 — Collecte de la couverture par scénario

Le script scripts/run_per_scenario_coverage.sh exécute chaque scénario Cucumber dans une JVM distincte, puis renomme le fichier target/jacoco.exec produit en sc_NNN.exec. 

./scripts/run_per_scenario_coverage.sh --no-pom-changes /chemin/projet

Pour chaque scénario, la commande Maven exécutée est de la forme :

mvn test \
  -Dcucumber.filter.name="^Nom du scenario$" \
  -DargLine="-javaagent:jacocoagent.jar=destfile=target/jacoco.exec,append=false"

Sortie produite :

coverage-per-scenario/
├── sc_001.exec        // couverture binaire JaCoCo
├── sc_002.exec
├── ...
└── index.txt          // sc_001|Should be authenticated
                       // sc_002|Add bruce wayne user
                       // ...
Choix d'isolation — un fork JVM par scénario élimine la pollution d'état (caches statiques, sessions persistantes, base in-memory). Le coût est l'overhead de démarrage Maven, négligeable à l'échelle d'une évaluation académique.

3.3. Étape 2 — Construction de la map scénario → méthodes

La classe GroundTruthBuilder lit chaque .exec via l'API JaCoCo (ExecFileLoader + Analyzer), en croisant les probes activées avec le bytecode compilé. Pour chaque scénario, elle produit la liste des méthodes effectivement exécutées. 

java -cp target/rts-llm-1.0-SNAPSHOT.jar com.rts.llm.GroundTruthBuilder build \
    /chemin/projet/coverage-per-scenario \
    /chemin/projet/target/classes \
    /chemin/projet/target/test-classes \
    coverage.json

Format de l'identifiant méthode : FQN.NomClasse.nomMéthode, identique à celui produit par StaticAnalyzer, pour permettre l'intersection avec les méthodes modifiées extraites du diff.

Normalisations appliquées sur la sortie JaCoCo :

  • com/example/Foo → com.example.Foo (slash → point)
  • Outer$Inner → Outer.Inner (classes imbriquées)
  • lambda$importFrom$0 → importFrom (repli des lambdas sur leur méthode englobante)
  • <init>, <clinit>, access$N : ignorés (non trackés par StaticAnalyzer)

Extrait de coverage.json :

{
  "sc_001": {
    "name": "Should be authenticated",
    "methods": [
      "fr.redfroggy.bdd.restapi.user.UserController.getAll",
      "fr.redfroggy.bdd.restapi.glue.AbstractBddStepDefinition.setUp",
      ...
    ]
  },
  ...
}

3.4. Étape 3 — Extraction des méthodes modifiées par le diff

Pour un diff Git donné (git diff C^ C), SymbolExtractor :

  1. parse les en-têtes de hunks (@@ -X +Y,Z @@) pour récupérer les plages de lignes modifiées dans chaque fichier ;
  2. croise ces plages avec methodRangesByFile (calculé par StaticAnalyzer via JavaParser) ;
  3. retourne la liste des méthodes dont la plage chevauche au moins un hunk.

3.5. Étape 4 — Intersection finale

La vérité terrain pour le diff est obtenue par intersection :

for (Map.Entry<String, ScenarioCoverage> e : coverageMap.entrySet()) {
    if (!Collections.disjoint(e.getValue().methods, modifiedMethods)) {
        groundTruth.add(e.getKey());
    }
}

3.6. Boucle sur l'historique Git

Le script scripts/eval_history.sh automatise les étapes 1 à 4 sur une plage de commits :

Pour chaque commit C dans la plage : 1. checkout C^ (parent) <- état AVANT le diff 2. mvn test-compile <- bytecode pour analyse JaCoCo 3. run_per_scenario_coverage.sh <- un .exec par scenario 4. GroundTruthBuilder build <- coverage.json 5. checkout C 6. GroundTruthBuilder eval <- calcule G pour le diff C^ -> C 7. agrege dans un CSV
Important — la couverture est collectée sur le parent du commit (C^), c'est-à-dire l'état du code avant les modifications. C'est nécessaire pour que les noms et plages des méthodes correspondent à celles que le diff référence en -.

4. Métriques d'évaluation

Pour un commit donné, on compare deux ensembles :

  • G — la vérité terrain (scénarios réellement impactés)
  • S — la sélection produite par RTS-LLM
Sigle Nom Définition Interprétation
TP True Positive |S ∩ G| Sélectionnés et impactés — bonne décision
FP False Positive |S \ G| Sélectionnés mais pas impactés — temps perdu
FN False Negative |G \ S| Impactés mais oubliés — danger
Précision = TP / (TP + FP)
Rappel = TP / (TP + FN)
F1 = 2 · P · R / (P + R)

Pour un outil RTS, le rappel est l'indicateur prioritaire : un FN signifie qu'un test impacté n'est pas exécuté, donc qu'un bug peut atteindre la production sans alerte. Un FP n'est qu'un test exécuté inutilement.

5. Limitations connues

5.1. Surcharges de méthodes (overloads)

Les identifiants utilisés (FQN.Class.method) n'incluent pas la signature. Si une classe possède plusieurs surcharges et que seule l'une d'entre elles est modifiée, la vérité terrain marquera comme TP des scénarios qui n'utilisaient en réalité qu'une autre surcharge.

Effet : surestime gt_size. Faible impact sur les projets sans surcharges intensives.

5.2. Méthodes ajoutées par le diff

La couverture est mesurée à C^. Si C introduit une méthode neuve, aucun scénario n'a pu la « couvrir » dans la map produite à C^. La vérité terrain ignore donc les scénarios qui, après C, utiliseront cette méthode.

Effet : sous-estime gt_size, biaise contre RTS-LLM s'il a (correctement) sélectionné ces scénarios.

Atténuation possible : doubler le coût de collecte en mesurant aussi la couverture à C et en unissant les deux maps.

5.3. Classes anonymes

JaCoCo nomme une méthode dans une classe anonyme EnclosingClass$1.method. Après normalisation : EnclosingClass.1.method. JavaParser, lui, ne génère pas de ClassOrInterfaceDeclaration pour les classes anonymes : StaticAnalyzer ne produit donc aucun identifiant correspondant. Méthode invisible des deux côtés.

5.4. Tests qui échouent

Le script garde le fichier .exec même si mvn test retourne un code d'erreur. Si un scénario plante avant d'atteindre le code modifié (setup cassé, dépendance externe indisponible), sa couverture est incomplète → sous-estime gt_size.

5.5. Renommages

Un renommage de méthode dans C apparaît comme une suppression dans C^ et un ajout dans C. SymbolExtractor peut ne pas corréler les deux. Heureusement, un renommage pur n'altère pas le comportement observable : aucun scénario ne devrait être impacté de toute façon.

6. Sanity checks de validation

Avant de se fier aux chiffres, quatre vérifications rapides :

CheckProcédureAttendu
Diff trivial Commit qui ne touche que des commentaires gt_size = 0
Diff massif Commit qui modifie une classe centrale gt_size proche du total
Présence dans la map Vérifier qu'au moins une méthode du diff figure dans coverage.json grep "NomMéthode" coverage.json retourne quelque chose
Cross-check avant/après Sur 2-3 commits : mvn test à C^ puis à C, comparer les scénarios qui changent de résultat L'ensemble obtenu doit être inclus dans G
Recommandation — exécuter le quatrième check sur un échantillon est l'argument le plus solide à donner dans un mémoire pour défendre la fiabilité de la vérité terrain.

7. Bibliographie & rapport au mémoire

La méthodologie ici décrite est une instance du paradigme safe RTS par couverture dynamique, formalisé notamment par :

  • Rothermel & Harrold (1997), A Safe, Efficient Regression Test Selection Technique, ACM TOSEM.
  • Gligoric, Eloussi & Marinov (2015), Practical Regression Test Selection with Dynamic File Dependencies — outil Ekstazi.
  • Legunsen et al. (2017), An Extensive Study of Static Regression Test Selection in Modern Software Evolution — outil STARTS.

Ces travaux utilisent JaCoCo (ou des instrumentations équivalentes) comme oracle, exactement de la manière décrite dans ce document. Ils servent de référence à la fois baseline (rappel = 1 par construction quand l'instrumentation est complète) et justification méthodologique de l'approche.

Pour une évaluation de RTS-LLM, la vérité terrain est l'objet de comparaison ; les métriques de précision, rappel et F1 publiées dans le mémoire sont la projection directe des intersections S ∩ G, S \ G, G \ S calculées par GroundTruthBuilder.evaluate().
